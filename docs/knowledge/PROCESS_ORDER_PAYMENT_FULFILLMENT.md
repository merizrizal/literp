# Order, Payment, and Fulfillment Process (POS Channel)

## Overview
This document describes the business process and data transitions for POS order lifecycle in Literp.

Core principle:
- Sales captures intent.
- Inventory records physical state changes.
- POS acts as a channel adapter.

## Process Scope

### In-scope data model
- `sales_order`
- `sales_order_line`
- `inventory_reservation`
- `payment`
- `inventory_movement`

### Related master data prerequisites
- `unit_of_measure`
- `product`
- `product_variant`
- `location`

## Preconditions
Before a POS order can be processed:
- Product catalog is available and active.
- Selling location exists.
- Pricing is available at channel/application layer.

No reservation, payment, or movement transaction is created in this phase.

## Phase 1: Create Draft Order

### Business intent
Start a new customer transaction.

### Data perspective
- Create `sales_order` with:
  - `status = DRAFT`
  - sales channel context
  - `total_amount = 0`

### Inventory impact
- None.

## Phase 2: Add Order Lines

### Business intent
Capture selected items and commercial values.

### Data perspective
- Create `sales_order_line` entries with ordered quantity and unit price snapshot.
- Compute each line total.
- Recalculate order total on `sales_order.total_amount`.

### Inventory impact
- None.

### Business rule
- Draft phase is editable and reversible.

## Phase 3: Confirm Order and Reserve Stock

### Business intent
Commit customer intent and reserve stock.

### Data perspective
- Transition order state from `DRAFT` to `CONFIRMED`.
- Create `inventory_reservation` per line with `status = RESERVED`.
- Align line state to reserved lifecycle.

### Inventory impact
- No physical movement yet.
- Stock is earmarked through reservations.

## Phase 4: Capture Payment

### Business intent
Record incoming customer payment.

### Data perspective
- Create payment records linked to the order.
- Update captured amount balance against order total.

### Inventory impact
- None.

## Phase 5: Fulfillment

### Business intent
Execute physical hand-over and complete stock effect.

### Data perspective
- Validate order readiness for fulfillment.
- Create `inventory_movement` entries with sales reference metadata.
- Mark reservations and lines as fulfilled.
- Transition order to fulfilled state when all lines complete.

### Inventory impact
- This is the authoritative stock deduction event.

## Phase 6: Cancellation (Pre-Fulfillment)

### Business intent
Abort order prior to physical completion.

### Data perspective
- Transition order to cancelled state.
- Cancel non-fulfilled lines.
- Cancel active reservations.

### Inventory impact
- No movement entry is generated by cancellation itself.

## State Transition Model

### Sales order
- `DRAFT -> CONFIRMED -> FULFILLED`
- `DRAFT -> CANCELLED`
- `CONFIRMED -> CANCELLED` (subject to payment policy)

### Sales order line
- `PENDING -> RESERVED -> FULFILLED`
- `PENDING/RESERVED -> CANCELLED`

### Inventory reservation
- `RESERVED -> FULFILLED`
- `RESERVED -> CANCELLED`

### Payment
- Captured payments contribute to settlement tracking.
- Extended lifecycle statuses remain available at schema level.

## Process Matrix (Domain/Data View)

| Phase | `sales_order` | `sales_order_line` | `inventory_reservation` | `payment` | `inventory_movement` |
|---|---|---|---|---|---|
| Draft | Create draft order | None | None | None | None |
| Add lines | Update order total | Create/update lines | None | None | None |
| Confirm | Set confirmed | Move to reserved lifecycle | Create reserved rows | None | None |
| Payment | No mandatory state change | No mandatory change | No mandatory change | Create payment rows | None |
| Fulfill | Set fulfilled | Set fulfilled quantities/status | Set fulfilled | No mandatory change | Create OUT movement rows |
| Cancel | Set cancelled | Cancel non-fulfilled lines | Cancel reserved rows | Governed by payment policy | None |

## Implementation Notes
- Multi-step state changes should be treated as atomic business transactions.
- Reservation and movement records are critical for auditability.
- Receipt generation and refund orchestration are separate extensions of this process.
