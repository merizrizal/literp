## Architectural Design Specification: Phase 03 Order Command Hardening, Idempotency, Auditability, and Stock Availability

**Source:** `docs/implementation-plan/03-order-inventory-flow.md`, tasks `03.1`, `03.2`, and `03.3`

**Goal:** Harden the POS-first order-to-fulfillment workflow by making multi-table order commands atomic, retried commands safe, order state transitions auditable, order numbers robust under concurrency, and stock availability queryable/enforced before reservation.

---

### I. Overview and Contract

This ADS covers three ordered Phase 03 tasks:

- `03.1 Transactional Order Commands`
- `03.2 Command Idempotency And Order Auditability`
- `03.3 Stock Availability And Reservation Policy`

The work must preserve the phase domain rule from the plan: **sales captures intent, inventory records physical state changes**. Reservations are earmarks; fulfillment remains the authoritative stock deduction event.

Primary existing API surface:

- `POST /orders/{salesOrderId}/lines`
- `POST /orders/{salesOrderId}/confirm`
- `POST /orders/{salesOrderId}/payments`
- `POST /orders/{salesOrderId}/fulfill`
- `POST /orders/{salesOrderId}/cancel`
- Proposed stock endpoints for `03.3`:
  - **Endpoint Contract (Conceptual):** `GET /stock/current?productId=<id>&locationId=<id>` or an equivalent REST path confirmed during implementation.
  - **Endpoint Contract (Conceptual):** `GET /stock/available?productId=<id>&locationId=<id>` or an equivalent REST path confirmed during implementation.

Existing concrete service contracts:

**Function Signature Contract (Concrete):** `OrderProcessService.addSalesOrderLine(String salesOrderId, String productId, String sku, String quantityOrdered, String unitPrice): Future<JsonObject>`

### IV. Step-by-Step Procedure / Execution Flow

#### 03.1 Transactional Order Commands

1. Re-confirm transaction coverage in `OrderProcessRepository`.
2. Keep `addSalesOrderLine` transaction behavior if the existing `inTransaction` wrapper is correct; add rollback coverage for insert-line + recalc-total behavior.
3. Convert `confirmSalesOrder` to execute all reads/writes through one transaction connection:
   - read order and status
   - read lines
   - optionally check stock availability from `03.3` once available
   - insert reservations
   - update line statuses
   - update order status
4. Convert `fulfillSalesOrder` to execute payment guard, line reads, movement inserts, reservation updates, line updates, and order status update in one transaction.
5. Convert `cancelSalesOrder` to execute captured-payment guard, order update, line update, and reservation update in one transaction.
6. Add rollback tests that trigger at least one failure after an earlier write in each command path.

#### 03.2 Command Idempotency And Order Auditability

1. Decide idempotency input contract:
   - Proposed: HTTP `Idempotency-Key` header for commands.
   - Alternative: request body field, but this changes request schemas and proxy methods differently.
2. Add persistence for idempotency records or command-result fingerprints.
3. Make confirm, payment, fulfill, and cancel command retry behavior safe:
   - same command + same key returns the original successful result or a stable already-applied response
   - same key + conflicting payload returns conflict
   - no duplicate reservation/payment/movement rows are created on retry
4. Replace app-only order number generation with database-backed generation:
   - Preferred: PostgreSQL sequence or table-backed counter combined with order date/channel formatting.
   - Keep `sales_order.order_number` unique as final guard.
5. Add order state-transition auditability:
   - Proposed: `sales_order_event` table with event ID, order ID, event type, previous status, new status, command key, actor/createdBy where available, notes/reason, created_at.
   - At minimum, events for create draft, add line, confirm, payment captured, fulfill, cancel.
6. Document client behavior for retries and idempotency.

#### 03.3 Stock Availability And Reservation Policy

1. Add current stock query calculation from `inventory_movement`.
2. Add reserved quantity calculation from active `inventory_reservation` rows.
3. Add available stock calculation: current stock minus active reservations.
4. Expose current and available stock APIs after route/service/repository contracts are confirmed.
5. Enforce available stock before reservation creation in `confirmSalesOrder`.
6. Decide oversell policy:
   - Proposed default: no oversell for all channels in Phase 03.
   - If channel-specific oversell is needed, document policy and implement behind an explicit database/config policy rather than hard-coded assumptions.

### V. Failure Modes and Resilience

| Stage | Failure Mode | Agent/System Action | Next State/Error Report |
|---|---|---|---|
| Add line | Line insert succeeds but total recalculation fails | Transaction rolls back all writes | Order remains `DRAFT`; no new line persists; 500 or mapped DB error |
| Confirm | Reservation insert succeeds for one line then later line/order update fails | Transaction rolls back reservations and status updates | Order remains `DRAFT`; no partial `RESERVED` lines/reservations |
| Confirm | Available stock is below requested reservation quantity | Reject before reservation insert | Order remains `DRAFT`; 409 conflict with insufficient stock message |
| Confirm retry | Same confirm command sent twice | Idempotency returns original/stable confirmed response; no duplicate reservations | Order remains `CONFIRMED`; reservation count unchanged |
| Payment retry | Client retries captured payment after timeout | Idempotency returns original payment result; no duplicate payment | Captured total unchanged after first success |
| Fulfill | Movement insert succeeds then reservation/line/order update fails | Transaction rolls back movement and related updates | Order remains `CONFIRMED`; no orphan `OUT` movement |
| Fulfill retry | Same fulfill command sent twice | Idempotency returns original/stable fulfilled response; no duplicate movements | Order remains `FULFILLED`; movement count unchanged |
| Cancel | Order update succeeds but line/reservation update fails | Transaction rolls back all updates | Previous order/line/reservation statuses remain |
| Cancel retry | Same cancel command sent after prior success | Return stable already-cancelled/idempotent response | Order remains `CANCELLED`; audit does not duplicate incorrectly |
| Order number | Concurrent draft creation generates same number | Database-backed generation and unique constraint prevent duplicates | One or both create attempts get unique generated numbers; unexpected collision surfaces as retryable error |
| Audit write | State transition succeeds but audit insert fails | If audit is part of command transaction, roll back command; if audit is outbox-style, record retryable audit failure | Preferred: no state transition without audit for Phase 03 integrity |
| Stock query | Movement semantics ambiguous for `OUT` rows due `to_location_id = from_location_id` workaround | Implement query with documented current behavior and flag dependency on `03.4` | Tests pin current semantics; follow-up task may adjust query |

### VI. Security, Integrity, Idempotency, and Cleanup

- Use parameterized SQL only; preserve existing `Tuple`-based query binding.
- Treat idempotency keys as client-supplied identifiers, not authentication credentials.
- Avoid logging raw payment references if they may contain sensitive gateway data.
- Enforce uniqueness at the database level for order numbers and idempotent command records.
- Keep all command state changes and audit event writes inside the same transaction unless an explicit outbox pattern is introduced.
- Maintain immutable `inventory_movement` semantics; do not update movement rows to repair command failures.
- Cleanup should happen through transaction rollback, not compensating deletes, for command failures.
- Idempotency record cleanup/retention policy must be documented; proposed default is retain long enough to cover client retry windows and order audit needs.
- Stock reservation enforcement must prevent negative available stock unless an explicit oversell policy says otherwise.
